# Shorter-tick + fleet-split — consolidated option menu (Round 1 of the council, 2026-07-08)

Goal (Kiesh): continuous, SMOOTH trading with as many bots as possible at a LOW tick. Proposal: shorter
ticks (~250ms) + split the ~20k fleet into N passes (~5k/tick). Open Qs he raised: split advanced orders
out? dynamically choose N (scaler)? overload → scaler cap-down. Generated by 3 architects (minimal /
adaptive / structural). **Round 2 debates THIS menu → a recommendation.** CK=0 sacred; replay valued;
validate ON PROD (docker inflates per-statement latency ⇒ local absolute numbers don't decide the floor).

## The shared baseline everyone starts from
**250ms tick + `Slots=4`** → `P = tick × N = 1s` per-bot cadence UNCHANGED (a bot still acts ~1×/s, just
phase-shifted). Config-only (both `TradeInterval` + `Bots:Staggering:{Enabled,Slots}` are wired; stagger
`StaggerDue(id,tick,N)` is a pure, deterministic, CK-clean count-split). This IS the core smoothness lever:
4 smaller fill-bursts/sec matching the chart's existing 250ms `QuoteRegistry` drain → a smoother tape. The
rest of the menu is variations on **overload handling**, **advanced-order handling**, and **the scan floor**.

## Axis A — OVERLOAD control (what gives when a tick can't finish in 250ms)
- **A0 Cap-only** — the existing scaler cuts `ActiveBotCap`. One controller, simplest, most deterministic;
  sheds bots under load. (= Kiesh's "if it can't handle it, the scaler goes down.")
- **A1 Elastic period** — `SelfCorrectingDelay` stretches the tick (250→~350ms) under load, HOLDS the fleet.
  Best for "as many bots as possible"; weakest replay (period becomes load-dependent).
- **A2 Dynamic N** — the scaler controls `Slots`; a `scanFrac` gate raises N when the *actionable* span
  dominates, else falls through to the cap cut when the fixed scan dominates. Direct answer to "dynamic N"
  + "overload→cap down". Replay = log the N-timeline (no worse than the dynamic cap we already ship).
- **A3 Single-controller priority ladder** — one supervisor holds a token; only one actuator moves per
  window: stretch → cap → N(at regime boundary). Strongest anti-hunting (three actuators, ONE controller).
- **A4 Cascaded two-timescale** — frequency-separated loops: fast=period (~4Hz), mid=cap (~0.25Hz),
  slow=N (~epoch). Textbook-stable (each loop ≥10× slower than the one above); most machinery.

## Axis B — ADVANCED-order handling
- **B0 Ride the same N** — simplest; advanced compute stays inline in the staggered walk.
- **B1 Coarser cadence for advanced + cohorts** — run advanced/arb/mm/rotator/jump only every 4th tick
  (`TickId % 4 == 0`) / a coarser `AdvSlots`. Keeps the EXPENSIVE settle-group work at 1× while spraying
  the cheap plain-limit fills 4× for smoothness. Small `TickId%k` guards.
- **B2 Decoupled advanced-SCAN** — pull stop/trailing/bracket *evaluation* out of the free-will gates so a
  stop fires the sub-tick price CROSSES it (250ms latency), independent of the bot's trade cadence. **NOT a
  perf win (docker ghost — adv is cheap on prod); it's a STOP-LATENCY / REALISM axis** (crisper stop
  cascades). Moderate CK scrutiny (more triggers/reservations churn; auditor still terminal).
- **B3 Multi-rate scheduler** — a general pass table `{intervalTicks, slots, phase}` per pass; the superset
  of B1/B2; highest complexity (refactors the loop spine); marginal prod value beyond B1/B2.

## Axis C — THE SCAN FLOOR (the unifying constraint every option is really a stance on)
`CollectPendingOrders` iterates ALL ~20k bots EVERY tick — stagger gates ACTION, not iteration (the per-bot
burst/RNG bookkeeping is a per-tick contract). So at 250ms the fixed 20k scan runs **4×/sec**. Splitting
only thins the *actionable* (decision+batch) span, NOT the scan. This sets the hard tick floor.
- **★ C1 The floor-breaker** — index bots by slot so the loop iterates only the ~`cap/N` DUE set
  (O(cap/N), not O(20k)). Turns short ticks from "4× the scan" into cheap, and MULTIPLIES the payoff of
  dynamic-N (A2/A4). BUT it's an engine change touching the per-tick-per-bot RNG bookkeeping contract →
  determinism-sensitive (must preserve or re-seed the RNG-stream advance). Bigger, riskier, higher-leverage.

## Honest framings the debate must weigh
1. **Do we even need more than the baseline?** Architect C: for "smooth chart + many bots," the by-count
   baseline (250ms+Slots=4+cap-only) likely gets MOST of it; the rest may be unnecessary complexity.
2. **Splitting advanced for PERF is a docker ghost** — adv is cheap on prod + the by-count split already
   cuts fleet load. Split advanced ONLY for stop-latency/realism (B2), a different axis than the goal.
3. **The scan floor decides how short the tick can go** — and it can only be measured ON PROD (old profile:
   collect ~4-18ms ⇒ 4×/sec is affordable; if grown, C1 is needed). Don't decide it on docker numbers.
4. **Kiesh's two wants pull differently under overload:** "low tick / continuous" ⇒ keep tick short; "as
   many bots as possible" ⇒ hold the fleet. The fork is A0 (shed bots) vs A1 (stretch tick) vs A2 (raise N).
5. CK=0 is safe across ALL options (single-threaded). Replay = log whichever control timeline moves (cap/N/period).

## The forks Round 2 must resolve
(1) Baseline-only, or add control? (2) Overload = shed-cap / stretch-period / dynamic-N — which, no hunting?
(3) Advanced = same-N / coarser-cadence / decoupled-scan? (4) Is the scan-floor-breaker (C1) worth the
engine change now, or defer until a prod measurement proves the floor binds?
