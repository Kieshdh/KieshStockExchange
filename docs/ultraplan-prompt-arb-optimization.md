# Ultraplan — Arbitrage subsystem perf optimization (the ~100 ms/tick arb phase)

**Goal:** cut the arbitrage tick phase's fixed ~100 ms/tick in the commit-bound, single-threaded bot loop, via a **sequenced set of flag-gated, default-off, byte-identical-off, CK-soak-gated levers** — so every step is freeze-safe (nothing changes until validated + flipped, exactly like the just-shipped buy-stop / short-open batches). Conservation (CK=0) is sacrosanct. Do **not** touch the FX house economics.

## ✅ Phase 0 RESULT (measured 2026-07-06) — the flag-flip is largely SPENT → Phases 1–3 ARE the lever
A/B (baseline vs `BatchLegs`+`GroupCommit`, 45 min, **CK CLEAN both**): arb-ms/tick **190.8 → 114.7** and commits/sec **24.6 → 21.1**. BUT the two parallel arms settled at **unequal caps (881 vs 628)**, so most of that arb-ms drop is the lever running *thinner books → faster scan*, not the batching — and the tiny commit drop confirms the **leg-batch is largely SPENT** (arb legs span *different* stocks, so group-commit doesn't coalesce them — the arc's prior "batch the entry is spent" finding). A pinned-cap clean confirmation is running to settle the exact commit Δ.
**⇒ Do NOT lead with the flag-flip.** The real, cap-independent reclaim is **Phase 2 (event-driven scan** — attacks the ~190 ms/tick scan directly**) + Phase 1 (batch the unbatched FX-rebalance `ConvertAsync` commits)**. Design Phases 1–3 with the event-scan as the primary lever; treat the flag-flip as a cheap default-on baseline, not the win.

## ★ Diagnosis (code-grounded from a sweep — build on this, don't re-derive)
The `_phArbUs` phase is **EXECUTION-dominated (DB commits), not the in-memory scan.** ~5 arb bots (seeded UserIds 20003–20007, `ExcelSeedService.cs:345-383`), each acting per `DecisionInterval` (default 1 s, `AIUser.cs:53`) does, in `ArbitrageDecisionService.RunAsync` (`:71`, bots iterated ascending AiUserId `:84`):
1. **Flatten** (`TryFlattenAsync :296`) — loops **all** cross-listed candidates, reads position + both books, market-sells residual inventory.
2. **2-leg round-trip** (`TryRoundTripAsync :226-249`) — leg-1 `PlaceTrueMarketBuyOrderAsync :233` → **its own root commit/fsync**; leg-2 `PlaceTrueMarketSellOrderAsync :246` → **a second root commit**. Opportunity scan = `CollectOppsAsync :174` (reads both USD+EUR top-of-book per candidate per bot every tick via `ReadTopAsync`/`GetAsync :148`, gap in `EvaluateDirection :160` off `_fxRates.GetBidAsk`, keep ≥ `MinArbitrageRatePrc :193`, `PickWeighted` ∝ rate² `:369`).
3. **FX rebalance** (`MaybeRebalanceAsync :323` → `_portfolio.ConvertAsync :361`) on the bot's `ConversionCadenceSeconds` clock — pays `ConvertSpread` into the house (20002).

**Cost model:** each arb leg ≈ **5 ms/fsync** (its own tx) vs ~0.6 ms plain. 5 bots × 2 legs ≈ **50 ms**, + the **unbatched per-bot `ConvertAsync` rebalance commits** (now firing every cadence since `ConversionSkewBand` was lowered to 0.03), + occasional flatten sells. The scan is real but secondary — and mostly wasted, because the **FX mid only re-ticks once per 60 s** (`FxRateService.cs:12`, `Tick :103`).

## ★★ Council corrections + guardrails (READ FIRST — they correct the naive framing)
- **`Db:GroupCommit:Enabled` is ALREADY BUILT + baked** (the group-commit slice; on in current soaks). So "batch the legs" is a **two-flag flip** (`Bots:Arbitrage:BatchLegs` + `Db:GroupCommit:Enabled`), **not** a settlement rebuild. The batched path already exists — `ExecuteBatchedEntriesAsync :255-292` → `OrderEntryService.cs:757-817` (`PlaceAndMatchBatchAsync`, the same amortized match+settle path as the shipped short-open batch `:722`).
- **`BatchLegs` was disabled as a MEASURED NO-WIN, not a bug** — the arc found "batching the arb ENTRY is spent" because the match+settle cost is per-`(stockId,currency)` **group**, not the entry-insert; batching orders across *different* stocks doesn't cut the group count. **BUT** that test predates pairing it with GroupCommit's per-currency-shard coalescing (per-stock groups → one root commit per currency shard). ⇒ **Phase 0 re-measures `BatchLegs`+`GroupCommit` TOGETHER — a genuinely new combination, not a re-run of a dead test.** (Outsider's "why was it off?" is answered: no-win, not a correctness bug.)
- **The 50 ms ≠ 100 ms gap:** legs-batching caps at ~half the phase; the unbatched `ConvertAsync` rebalances + flatten sells are the OTHER half → **maximal** optimization needs the rebalance batch too.
- **DON'T narrow `ConvertSpread`** (`FxRateService.cs:19,55-61`) — it's the house's revenue + the cross-currency coupling floor. Off-limits as a perf move.
- **Preserve** the `ValueDrainCeilingPct` throttle (`BotEconomyTelemetry.ArbThrottleEngaged :30,161-166`, read cheaply at `ArbitrageDecisionService :75`), the house (20002) economics, and `FxDeskTelemetry`.
- **Freeze-safe discipline:** every lever ships FLAG-GATED, DEFAULT-OFF, BYTE-IDENTICAL off, with its OWN CK conservation soak before flip-on. Seed-determinism preserved (arb bots act in AiUserId order).

## What to design — sequenced by risk/value (each its OWN commit + CK soak)

**Phase 0 — MEASURE (no code).** Flip `BatchLegs=true` + `Db:GroupCommit:Enabled=true` together; `Bots:PhaseTimingSeconds>0`; 45-min CK soak on a short/arb-active fleet. Read the `BotPhase` `arb` ms + commits/sec (`AiTradeService.cs:1667`, `EngineCommitMetrics.RecordRootCommit`). Split scan-vs-exec empirically — the one uncertainty is the per-bot cross-listed watchlist size (seed-dependent). **This sizes Phases 1-3.** GATE: CK=0; record the arb-phase Δ. *If the flag-flip alone reclaims most of the ~50 ms leg cost, Phases 1-3 may be deferable — let the number decide.*

**Phase 1 — batch the FX-rebalance `ConvertAsync`** (`Bots:Arbitrage:BatchRebalance`, default-off). Collect the cohort's per-tick rebalance conversions and settle them in one amortized tx (mirror the leg/short-open batch). ⚠ **house-credit atomicity:** batching changes *when* the house (20002) is credited the spread — the per-conversion fund credit must stay conservation-exact (no ledger drift; watch `CK_Funds` + house-revenue accrual, not just aggregate CK).

**Phase 2 — event-driven arb scan** (`Bots:Arbitrage:EventDriven`, default-off) — the structural centerpiece. Replace the per-tick `CollectOppsAsync` fan-out with a **lock-free watcher mirroring `HostedServices/StopTriggerWatcher.cs`** (subscribe `IMarketDataService.QuoteUpdated :156`, `OnQuoteUpdated :209`): maintain a cross-listed `(stockId)→live FX-adjusted gap` map, react **only** when a cross-listed quote (or the 60 s FX re-tick) moves the gap past `MinArbitrageRatePrc`; the arb tick early-outs when nothing crossed. De-churns ~99 % of scans + naturally throttles how often the legs fire. **Pure read-path — no settlement change → low conservation risk.** GATE: round-trips still fire on real opps (behavior-equivalent), CK=0, arb-phase ms down.

**Phase 3 — minor caching** (fold in or its own commit): cache the **static** cross-listed set (`CrossListedWatchlist :135` re-runs `IsListedIn` ×2/candidate/bot/tick); gate `TryFlattenAsync :296` on a has-any-inventory flag (skip the all-candidates loop when the cohort is flat — round-trips net flat); hoist the per-60 s-constant FX bid/mid lookups to once per tick.

**OUT OF SCOPE (note for Kiesh — NOT this ultraplan):** scaling the arb cohort 5 → 20-30 (the Expansionist realism upside: tighter cross-currency spreads + a structural forcing-function against LLN herding). That's a **reseed + behavior change**, not perf — bundle into the final reseed. Note the synergy: once these levers land, arb marginal cost/bot drops sharply, so this perf work *enables* a cheap bigger cohort later.
